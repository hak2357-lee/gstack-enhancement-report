🔬
gstack이란?

Y Combinator CEO Garry Tan이 제작한 오픈소스 AI 기반 소프트웨어 개발 프레임워크. Claude Code를 "가상 엔지니어링 팀"으로 변환하는 15개 이상의 전문 SKILL.md와 영속적 헤드리스 브라우저 자동화 도구로 구성된다. MIT 라이선스.

🎯
분석 목적

현재 시스템(.2nd_worktool)에 gstack의 검증된 패턴을 선별적으로 도입하여, 목표①: 완벽한 기획과 목표②: 완전 자율 실행을 달성한다. 무분별한 전체 도입이 아닌, 격차 분석 기반의 5개 핵심 보강을 제안한다.

⚡
핵심 주장: 60일 파트타임으로 600,000줄+

gstack 저자는 60일간 파트타임 작업으로 600,000줄 이상의 프로덕션 코드를 출력했으며, 일일 평균 10,000–20,000줄을 생성했다고 밝혔다. 본 보고서 섹션 7에서 압축률 데이터를 상세히 다룬다.

gstack 핵심 구성요소 분석

8개 핵심 구성요소 및 역할

📝
구성요소 01
SKILL.md 템플릿 시스템
.tmpl 파일에 플레이스홀더를 작성하면 gen-skill-docs.ts가 SKILL.md를 자동 생성한다. 코드와 문서 불일치를 근본적으로 차단하는 "코드에서 문서 생성" 패턴.
기반 인프라
🌐
구성요소 02
영속적 브라우저 데몬
Bun 컴파일 바이너리 + Playwright Chromium. 첫 호출 ~3초, 이후 호출 ~100ms. MCP 없이 0토큰 오버헤드로 브라우저 자동화를 제공한다.
브라우저 자동화
🔨
구성요소 03
25+ 전문 스킬
Think → Plan → Build → Review → Test → Ship → Reflect의 전체 스프린트 프로세스를 커버하는 역할 기반 전문 지시 파일. Claude를 CEO부터 보안 책임자까지 전환.
스프린트 파이프라인
📋
구성요소 04
3단계 기획 리뷰
/autoplan이 CEO 리뷰 → 디자인 리뷰 → 엔지니어링 리뷰를 순차 실행. 각 단계는 독립 스킬(/plan-ceo-review, /plan-design-review, /plan-eng-review)로 분리.
기획 품질
🛡
구성요소 05
안전 가드레일
/careful, /freeze, /guard — PreToolUse 훅 기반으로 파괴적 명령어 실행 전 경고 및 디렉토리 편집 하드 블록. 세션 스코프 보호 메커니즘.
안전성
🔗
구성요소 06
Preamble 시스템
모든 스킬에 {"{{PREAMBLE}}"} 블록이 자동 주입되어, 업데이트 체크·세션 추적·AskUserQuestion 표준·Search Before Building 원칙을 공통 적용한다.
공통 기반
🧪
구성요소 07
3-티어 테스트 전략
Tier 1 정적 검증(무료, <5초) → Tier 2 E2E via claude -p(~$3.85, ~20분) → Tier 3 LLM-as-judge(~$0.15, ~30초). diff 기반으로 관련 테스트만 자동 선택.
품질 보증
🤖
구성요소 08
멀티 AI 지원
Claude Code, Codex, Gemini, Kiro, Cursor를 단일 setup으로 지원. 각 에이전트에 맞는 스킬 경로를 자동 구성하며 Conductor로 병렬 워크스페이스 관리.
멀티 에이전트
3

현재 시스템 격차 분석

.2nd_worktool 프로젝트의 구조적 공백

⚠️
분석 전제

gstack 전체를 그대로 이식하는 것이 목표가 아니다. 현재 시스템의 구조적 공백 중 목표①(완벽한 기획)과 목표②(완전 자율 실행)을 저해하는 5개 핵심 격차를 선별했다.

📂

스킬 파일 미존재

planner가 참조하는 4개 스킬 파일(business-analyst.md, ux-researcher.md, solution-architect.md, qa-engineer.md)이 실제로 존재하지 않는다. 기획 단계 자체가 동작 불가 상태.

🔓

Hook 시스템 미구현 — 안전 가드레일 없음

settings.json에 PreToolUse 훅이 설정되어 있지 않다. 파괴적 명령어(rm -rf, DROP TABLE 등)에 대한 런타임 안전망이 전무하며, 자율 실행 시 치명적 위험.

🔀

자율 실행 메커니즘 의도만 명시 — 실제 구현 없음

CLAUDE.md 등에 자율 실행 의도가 기술되어 있으나, 실제 파이프라인(SDD→코드→테스트→커밋→알림)이 구현되지 않았다. 루프 제어, 타임아웃, 복구 로직 부재.

🗺

에이전트 간 라우팅/디스패치 규칙 없음

복수 에이전트 간 작업 분배 및 결과 전달 규칙이 정의되지 않았다. 어떤 에이전트가 어떤 조건에서 다음 에이전트를 호출하는지 명세가 없어, 다단계 자율 실행이 불가능하다.

📦

git 미설정

프로젝트 루트에 git 저장소가 초기화되어 있지 않다. 자율 실행 파이프라인의 최종 단계인 자동 커밋, 브랜치 관리, diff 기반 테스트 선택 등이 모두 작동 불가 상태.

4

5개 보강 제안

각 제안을 클릭하면 상세 내용을 확인할 수 있습니다

🎯
두 가지 목표 체계

목표①: 완벽한 기획 — 사업·기술·UX 관점의 다단계 리뷰로 기획 품질 극대화  |  목표②: 완전 자율 실행 — 승인된 기획서에서 출발해 코드·테스트·배포까지 사람 개입 최소화

01
다단계 기획 리뷰 체인
목표① 완벽한 기획 핵심 제안
▼

gstack의 /autoplan에서 영감을 받아, 단일 기획 리뷰를 사업 타당성 → 아키텍처 → UX 3단계 리뷰 체인으로 확장한다. 각 리뷰 단계마다 AskUserQuestion으로 사용자 피드백을 수집하여, 최종 기획서 품질을 보장한다.

/plan-business → 사용자 피드백 → /plan-technical → 사용자 피드백 → /plan-ux → /autoplan (통합)
💡
구현 포인트

/plan-business는 ROI·위험 요소·시장 적합성, /plan-technical은 아키텍처·DB 스키마·엣지케이스, /plan-ux는 사용자 흐름·접근성·인터랙션을 각각 전문 검토한다. /autoplan은 3개 리뷰 결과를 통합하여 최종 SDD를 산출한다.

사이드이펙트
  • 기획 소요 시간 2–3배 증가
  • 토큰 비용 약 3배 상승
  • 단순 작업에도 과도한 절차 적용 가능
보완책
  • /plan-quick(단일 리뷰)와 /plan-full(3단계) 선택지 제공
  • 작업 규모에 따른 자동 경로 선택 로직
  • 각 단계 결과를 파일로 저장하여 재사용
02
PreToolUse 훅 기반 안전 가드레일
목표②의 전제조건 Phase 1 즉시
▼

gstack의 /careful과 /freeze에서 직접 차용. settings.json에 PreToolUse 훅을 추가하여, 모든 Bash 명령어 실행 전 careful.py가 위험 패턴을 검사하고, freeze.py가 보호 디렉토리 편집을 차단한다. 자율 실행 파이프라인(제안 4) 운영의 필수 전제조건.

Bash 호출 요청 → PreToolUse 훅 트리거 → careful.py 검사 → freeze.py 검사 → 허용 / 경고 / 차단
🛡
차단 대상 패턴

rm -rf /, DROP TABLE, truncate, chmod 777, 보호 디렉토리 직접 편집, 프로덕션 DB 직접 연결 등 파괴적·비가역적 작업. 자율 실행 모드에서는 차단 대신 로그 기록으로 전환.

사이드이펙트
  • 모든 Bash 호출에 ~50ms 오버헤드
  • 정상 명령어 오탐(false positive) 가능
  • autonomous 모드에서 과도한 인터럽트
보완책
  • 신뢰 명령어 화이트리스트 관리
  • autonomous 모드 시 경고 → 로깅 전환
  • 훅 실행 결과를 별도 로그 파일에 기록
03
Preamble 시스템
기반 작업 Phase 1 즉시
▼

gstack의 {"{{PREAMBLE}}"} 시스템을 현재 구조에 맞게 이식한다. ~/.claude/templates/preamble.md에 공통 규칙을 정의하고, 모든 에이전트와 스킬이 이를 자동으로 포함한다. 개별 스킬마다 반복 작성하던 규칙을 단일 소스로 통합 관리.

📌
Preamble에 포함될 공통 규칙

Search Before Building 원칙 · Standard Output Envelope 형식 · AskUserQuestion 표준 · 한국어 응답 강제 · validate.sh 실행 의무 · Telegram 알림 트리거 조건 · 보안 금지 패턴

사이드이펙트
  • 프롬프트 토큰 ~500 증가 (매 호출)
  • preamble 변경 시 전체 스킬에 즉시 영향
  • 과도한 규칙 추가 시 지시 충돌 가능
보완책
  • validate.sh에 preamble 포함 여부 검증 추가
  • 규칙 변경 시 버전 태그 관리
  • 경량/전체 preamble 두 가지 프로파일 유지
04
자율 실행 파이프라인 /build-auto
목표② 완전 자율 실행 Phase 3
▼

승인된 SDD/BDD 문서를 입력으로 받아, 코드 생성 → behave 테스트 → LLM 검증 → git commit → 문서 갱신 → validate → Telegram 알림까지 완전 자동으로 실행하는 파이프라인. 각 단계에서 실패 시 최대 3회 자동 복구를 시도하고, 초과 시 강제 중단 후 알림을 발송한다.

SDD/BDD 입력 → 코드 생성 → behave 테스트 → LLM 검증 → git commit → 문서 갱신 → validate.sh → Telegram 알림
🔄
자동 복구 루프

각 Step 실패 시 최대 3회 재시도. 3회 초과 시 강제 중단 + Telegram 에러 알림 발송. Step별 타임아웃(기본 10분) 설정으로 무한 루프 방지. 모든 시도 결과는 logs/build-auto/에 타임스탬프와 함께 기록.

사이드이펙트
  • 잘못된 방향으로 3회 반복 위험
  • 테스트 미통과 상태에서 커밋 가능성
  • 장시간 실행으로 비용 예측 어려움
보완책
  • Step별 timeout 및 비용 상한선 설정
  • 모든 실행 로그 파일 기록 의무화
  • 가드레일(제안 2)과 반드시 병행 운영
05
3-티어 테스트 전략
목표② 지원 Phase 3
▼

gstack의 3-티어 테스트 체계를 현재 시스템에 이식한다. Tier 1(정적·무료)로 빠른 피드백, Tier 2(behave BDD·기존 자산 활용)로 기능 검증, Tier 3(statistical-validator·LLM 채점)로 품질 보증. Tier 3은 자율 실행 모드에서만 활성화하여 비용을 제어한다.

티어 도구 비용 속도 활성화 조건
Tier 1 정적 검증 validate.sh 강화 무료 <5초 항상
Tier 2 BDD behave (기존) 무료 1–10분 커밋 전
Tier 3 LLM 채점 statistical-validator ~$0.15/회 ~30초 자율 실행 모드만
사이드이펙트
  • Tier 3 LLM 채점 비용 추가 발생
  • 채점 기준 불명확 시 일관성 부족
  • Tier 2 테스트 장시간 실행 시 파이프라인 지연
보완책
  • Tier 3은 autonomous=true 모드에서만 활성화
  • 채점 rubric을 파일로 명시적 관리
  • Tier 2 타임아웃 설정으로 블로킹 방지
5

도입 로드맵

3단계 순차 도입 계획 — 안전성 → 기획 품질 → 자율 실행

📌
도입 순서 근거

안전 가드레일(제안 2)과 Preamble(제안 3)은 다른 모든 제안의 기반이 되므로 즉시 도입. 다단계 기획(제안 1)은 스킬 파일 제작이 필요하므로 1주 내. 자율 실행(제안 4)은 1–3이 완비된 후에만 안전하게 운영 가능.

1
Phase 1
즉시 도입 (Day 1–3)
안전성 확보 — 가드레일 + Preamble

다른 모든 보강의 전제 조건. git 초기화 및 기반 인프라 구성과 함께 진행.

제안 2: PreToolUse 가드레일 제안 3: Preamble 시스템 git init + 초기 커밋
결과: 파괴적 명령어 차단 활성화, 공통 규칙 단일 소스 확립
2
Phase 2
1주 내 (Day 4–7)
목표① 달성 — 다단계 기획 품질

4개 스킬 파일 제작 + 3단계 리뷰 체인 구현. Preamble 완비 후 진행.

제안 1: 다단계 기획 리뷰 체인 스킬 파일 4개 제작 /plan-quick 및 /plan-full
결과: 사업·기술·UX 3관점 기획 리뷰 완비, 목표① 달성
3
Phase 3
2주 내 (Day 8–14)
목표② 달성 — 3-티어 테스트 + 자율 실행

Phase 1, 2 완비 후 진행. 가드레일 없이는 자율 실행 파이프라인 운영 금지.

제안 5: 3-티어 테스트 전략 제안 4: /build-auto 파이프라인 statistical-validator 연동 Telegram 알림 연동
